SPEC-042 Async export pipeline for large datasets DRAFT

A functional brief defining what an asynchronous, resumable export pipeline must do before the architect commits it to a plan.

SPEC-042 created: 2026-05-30 · updated: 2026-06-22 · cycle 2
Problem Statement

Customers with large workspaces routinely request full data exports that exceed what the current synchronous endpoint can serve. Requests over roughly 200 MB time out at the gateway, leaving users with a failed download and no recourse but to retry — which fails the same way.

We need an export mechanism that decouples the request from the delivery: the user asks for an export, the system produces it in the background, and the user is notified when a stable, resumable artifact is ready to download. The experience must remain trustworthy even when an export takes minutes and spans transient infrastructure failures.

Functional Requirements 6 items
  • FR-1 Asynchronous request. A user can request an export and receive an immediate acknowledgement with a tracking ID; the export proceeds in the background.
  • FR-2 Progress visibility. The user can poll or subscribe to export status (queued, running, ready, failed) and see a rough percentage complete.
  • FR-3 Resumable delivery. The finished artifact supports HTTP range requests so an interrupted download resumes rather than restarts.
  • FR-4 Notification. On completion or failure the user is notified via their configured channel (in-app and email).
  • FR-5 Expiry. Ready artifacts remain downloadable for a bounded window, after which they are purged and the link returns a clear expired state.
  • FR-6 Idempotency. Re-requesting an identical export within the active window returns the existing artifact rather than recomputing it.
Constraints 4 items
  • C-1 Technical. Must run on the existing job queue (redis + worker pool); no new orchestration system this cycle.
  • C-2 Storage. Artifacts land in the existing object store; no schema migrations to the primary database.
  • C-3 Time. Target a single-cycle delivery; defer multi-region replication to a follow-up.
  • C-4 Scope. CSV and JSON formats only; spreadsheet and archive bundles are out of scope.
Open Questions 3 unresolved
  • What is the acceptable upper bound on export latency before we surface a "still working" reassurance state to the user?
  • Should the expiry window be fixed system-wide, or configurable per workspace plan tier?
  • Do we need signed, single-use download URLs, or is an authenticated session sufficient for the first release?
