Architecture 1 finding
Info Second session-validation path duplicates auth/session.js logic ADR recommended
src/api/routes/users.js:5

Why it matters
router.use('/users', legacySessionMiddleware) wires a second, divergent admin-token check alongside the new createSession JWT flow. Two independent paths can decide whether a request is authenticated, and a future fix to one path is easy to forget in the other.

Suggested fix
Consolidate to a single exported session-validation middleware in auth/session.js and have every router depend on it.

ADR draft: Single source of truth for request session validation
Context: the diff introduces two independent ways to authenticate a request (JWT-based createSession vs. a legacy shared-secret header check). This is hard to reverse once routes depend on either path and affects every future route added to this router. Recommend picking one strategy and removing the other before merge.
Security 2 findings
Critical Hardcoded admin token committed to source
src/auth/session.js:3

Why it matters
ADMIN_TOKEN is a literal secret string checked into version control. Anyone with read access to the repository (including this PR's diff) can authenticate as admin against legacySessionMiddleware.

Suggested fix
Load ADMIN_TOKEN from a secret manager or environment variable at startup, rotate the leaked value immediately, and add a pre-commit secret scanner.

Medium Session token logged in plaintext
src/auth/session.js:7

Why it matters
console.log prints the freshly minted JWT alongside the userId. Anything with access to application logs (log aggregators, stdout capture, crash reports) can hijack the session.

Suggested fix
Remove the token from the log line, or log only a non-reversible identifier such as a hash of the token.

Bugs & Improvements 2 findings
High Missing await leaves user as a pending Promise
src/api/routes/users.js:8

Why it matters
db.users.findById returns a Promise but is not awaited, so user is the Promise object rather than the row. res.json(user) serializes Promise internals instead of the actual user — every request to this route is broken.

Suggested fix
Add await: const user = await db.users.findById(req.params.id);

Low Off-by-one in pagination skips the first page's results
src/api/routes/users.js:16

Why it matters
page starts at 1 (parseInt(req.query.page) || 1), but slice(page * pageSize, ...) treats page 1 as the second page, so the first pageSize users are silently never returned.

Suggested fix
Use (page - 1) * pageSize as the slice start.
